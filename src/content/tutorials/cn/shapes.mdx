## 概述
在 CroqTile 中，**形状（shape）**是一等公民。本节将介绍如何在 CroqTile 代码中编程式地处理形状。

## 一等公民：形状
CroqTile 的核心职责是管理数据搬运，这对高效组织与处理大规模数据集至关重要，尤其在机器学习和高性能计算场景中。然而，大多数 C++ 编程环境对数据的处理较为随意——要么当作扁平指针，要么当作层次化结构（数组），而没有与之关联形状的原生表示。

相比之下，CroqTile 通过要求**任何声明或使用的数据都必须关联形状**，来强化代码安全性并简化带形状数据的编程。这使得 CroqTile 将形状视为一等公民。

## 使用 `mdspan` 定义形状

在 CroqTile 中，可以使用 `mdspan` 关键字定义形状，它表示 **Multi-Dimensional Span（多维跨度）**。该关键字表示用于计算的多维数据。

可以如下显式定义 `mdspan` 变量：

```choreo
mdspan s0 : [7, 8]; // Defines a 2D shape with dimensions [7, 8]
mdspan<1> s1 : [3]; // Defines a 1D shape with dimensions [3]
```

在此示例中，开头的关键字 `mdspan` 表示声明 *mdspan* 变量，其后为用户提供的变量名。每个 *mdspan* 变量都由**初始化表达式**初始化，该表达式由方括号 `[]` 内以逗号分隔的整数值组成。紧随变量名之后的符号 `:` 引出*初始化表达式*。
因此，`s0` 被定义为二维形状，具有 `7` 行和 `8` 列；`s1` 则是一维形状，维度为 `3`。

还可以在 `mdspan` 关键字后使用 `<>` **可选地**指定 *mdspan* 变量的**秩（rank）**。若显式指定秩，会告知 CroqTile 编译器进行秩一致性检查。若秩与对应的*初始化表达式*不一致，则会在编译期报错。例如：

```choreo
mdspan<3> s2 : [64, 32]; // error: the rank of mdspan is inconsistent
```

除显式 mdspan 声明外，CroqTile 编译器也可根据*初始化表达式*推断 `mdspan` 变量类型，从而无需显式写出 `mdspan` 关键字：

```choreo
s3 : [7, 8, 9]
```

在此代码中，`s3` 是秩为 `3`、维度为 `7, 8, 9` 的 `mdspan`。由于 CroqTile **要求 mdspan 必须在声明时完成初始化**，在实践中更倾向使用类型推断写法。

## 推导 *mdspan*
在实际编码中，常常需要从已有形状推导新形状。例如，可能需要进行数据**分块（tiling）**或**阻塞（blocking）**，这就要求对形状的各维度做除法；或者需要对某些维度**填充（pad）**，即在形状维度上加上增量。

在 CroqTile 中，这类形状推导可以通过对 *mdspan* 进行算术运算轻松完成。以下代码展示了一个示例：

```choreo
shape : [128, 64]; // initial shape 
new-shape0 : shape [(0) / 2, (1) / 4, 1];  // tile and reshape: [1, 64, 16]
new-shape1 : shape [(1) + 2, (0) / 16];    // pad and reshape: [66, 8]
```

在此示例中，`new-shape0` 由 `shape` 推导而来：维度 0 除以 `2`，维度 1 除以 `4`。这在高层语义上对应*分块*操作。此外，代码还为 `new-shape` 增加了一个新维度。在高层语义中，该操作常被称为*重塑（reshape）*。

在 CroqTile 中，`new-shape0` 的定义等价于：

```
new-shape0: [shape(0) / 2, shape(1) / 4, 1];
```

这里，初始的 `shape` 以逐元素方式显式列在 `[]` 内，而不是在 `[]` 外单独指定。但与先前版本类似，在已有形状上使用**元素访问**操作（记为 `()`）来获取各维度的值。显然，这种方式需要更多代码，但结果相同。因此，先前版本可视为对新形状完整*初始化表达式*的*语法糖*。

在代码示例中，`new-shape1` 同样由 `shape` 推导：它对维度 1 做了 `2` 的填充，并在推导出的形状中交换了维度顺序。

此外，还可以将 `mdspan` 作为一个整体参与推导：

```choreo
shape : [32, 72]
new-shape0 : shape;  // [32, 72]
new-shape1 : shape + 1;  // [33, 73]
new-shape2 : shape / 4;  // [8, 18]
new-shape3 : [shape, 6]; // [32, 72, 6]
```

注意，对 *mdspan* 变量的算术运算是**按维度**应用的。因此，语句

`new-shape1 : shape + 1;`

等价于

`new-shape1 : shape [(0) + 1, (1) + 1];`

`new-shape3` 的推导定义表明：在 *mdspan 初始化表达式*中使用 *mdspan* 变量会产生**拼接（concatenation）**行为。因此，声明

`new-shape3 : [shape, 6];`

等价于

`new-shape3 : shape [(0), (1), 6];`

先前版本同样可以视为完整定义的*语法糖*。

## *mdspan* 的求值
至此，我们见到的 mdspan 均为常量取值，这在许多场景下已足够，因为高性能设备内核常需根据固定的输入数据形状进行精细调优。在这些情况下，`mdspan` 在**编译期**求值，其取值不会产生额外的运行时开销或存储开销。

然而，某些场景下构建内核需要**运行时形状**（部分维度在执行时才确定）。CroqTile 通过为 `mdspan` 提供**符号维度（Symbolic Dimensions）**来支持这一点，相关内容将在后文介绍。在此类场景中，可能需要在运行时对各维度取值求值。好在 CroqTile 会在进入 CroqTile 函数之后、在 CroqTile 生成的主机代码中立即完成该求值，启动开销可忽略不计。**因此，在一般情况下，程序员可以忽略与 *mdspan* 相关的开销**。
