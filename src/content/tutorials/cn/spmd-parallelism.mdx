## 概述
要充分利用并行硬件，在 CroqTile 中编写并行代码至关重要。本节将介绍如何使用 `parallel-by` 块在 CroqTile 中实现并行代码。


## 并行执行块
在 CroqTile 中，并行运行的代码被放入*并行执行块*，表示同一段代码有多个实例在同时执行。

### 基本语法：`parallel-by`

`parallel-by` 块的基本语法如下：

```choreo
parallel p by 6 {
  // SPMD (Single Program, Multiple Data) code
}
```
我们将 `{}` 内的代码称为 SPMD 风格的**并行执行块**，将代码的每个实例称为**并行线程**。
在本例中：

- **`parallel`**：该关键字启动*并行执行块*。
- **`p`**：**并行变量**。它是一个整数，用于标识当前*并行线程*。每个*并行线程*执行相同代码，但*并行变量* `p` 取值不同。
- **`by 6`**：表示共有 6 个*并行线程*。因此，*并行变量* `p` 在不同线程上取值范围为 `0` 到 `5`。

在某些场景下，程序员在简单并行构造中可能不需要显式的*并行变量*，可以省略：

```choreo
parallel by 2 { ... }
```

这会启动两个并行线程执行。但两个线程无法针对不同数据工作，而 SPMD 程序在数据并行处理中通常需要这一点。

### 多级并行

在 CroqTile 中可以定义多级并行。示例如下：

```choreo
parallel p by 2 {
  // parallel-level-0
  parallel q by 12 {
    // parallel-level-1
  }
}
```

该代码定义了两级并行，取值分别为 `2` 和 `12`，即总共启动 `2 x 12 = 24` 个并行线程。每个线程由 `p` 与 `q` 的唯一组合标识。在 CroqTile 中，将 `p` 所在层级称为*parallel-level-0*，将 `q` 所在层级称为*parallel-level-1*。熟悉 CUDA 的读者可将*parallel-level-0*对应 CUDA 的**网格（grid）**层级，*parallel-level-1*对应 CUDA 的**线程块（block）**层级。

多级并行的概念源于具有层次化内存的硬件架构。程序员可观察到的一个关键差异是**同步开销**。例如，在 CUDA 中，同一**线程块**内线程的同步开销远低于跨不同**网格**的线程。这种层次化设计促使程序员将最需要同步（分歧最大）的代码放在**线程块**层级，以提升整体性能。

在 CroqTile 语法中，可以在一行内定义多级并行：

```choreo
parallel p by 2, q by 12 { ... }
```

这与前一示例的并行化效果相同，使用**逗号分隔**的 *parallel-by* 表达式，且假定两级并行之间没有代码。

### 子级并行

诸如 *CUDA* 等目标平台允许将单级并行划分为最多三个子级。在 CroqTile 中，对此使用 `ituple` 与 `mdspan` 语法：

```choreo
parallel {px, py, pz} by [1, 1, 2] {
  // higher synchronization cost
  // ...
  parallel t_index = {qx, qy} by [3, 4] {
    // lower synchronization cost
    // ...
  }
  // ...
}
```

本例定义了两级并行。第一级（*level-0*）划分为三个子级，由 `{px, py, pz}` 形式的 `ituple` 标注；第二级（*level-1*）划分为两个子级，由 `{qx, qy}` 形式的 `ituple` 标注，或整体记为 `t_index`。对每个子级，并行数量与 `mdspan` 中对应维度的取值一致。

子级的概念来自 *CUDA* 的 GPU 负载管理，其中常涉及二维或三维数据。注意 `ituple` 中**第一个**元素表示**最低有效**的并行变量。CroqTile 采用这一顺序是为了与 *CUDA* 开发者的习惯一致，而非出于内部一致性考虑。

### 有界变量

在 `parallel-by` 块中，`parallel` 关键字之后定义的变量称为**有界变量（Bounded Variables）**。具体而言：

- 若变量为*整数*，则称为**有界整数（bounded integer）**。
- 若变量为*ituple*，则称为**有界 ituple（bounded ituple）**。

该术语表示变量不仅有取值，还有明确的界限。例如，在语句 `parallel p by 6` 中，`p` 是*有界整数*，界限为 `[0, 6)`，其中 `6` 为不包含的上界。

有界变量的概念将在后续章节进一步展开，并详细说明其用法。

## 含义与影响

### 异构性：转译为内核启动

对于 *CUDA/Cute* 等目标，`parallel-by` 块不仅指定并行线程数量，还借助 CroqTile 管理异构性的能力，划定了主机代码与设备代码的边界。具体而言：

- **`parallel-by` 块之外**的代码转译为**主机代码**。
- **块之内**的代码转译为**设备代码**，如下所示：

```choreo
__co__ void foo(...) {
  // Generate host code
  // ...
  parallel p by 6 { // Kernel launch
    // Generate device code
    // ...
  }
  // Generate host code
  // ...
}
```

熟悉 *CUDA* 的读者可将此类比为在 `parallel-by` 语句处发生**内核启动（Kernel Launch）**。不过 CroqTile 用统一的 SPMD 并行模型抽象了这些细节，使开发者能专注于算法，而无需操心底层异构细节。

### 存储说明符的限制

由于 CroqTile 中的 `parallel-by` 构造与硬件相关（包括异构性与内存层次），它受目标平台约束。

例如，对于 *CUDA/Cute* 目标，在*并行执行块*内，*spanned* 缓冲区必须用适当的存储说明符标注。这是因为目标平台仅允许设备代码分配*暂存内存*（如 *shared* 与 *local* 内存）。反之，*并行执行块*之外的代码只能声明*全局*或默认主机内存，由主机代码管理。

此外，*shared* 与 *local* 内存类型的生命周期仅限于内核启动期间，因此在*并行执行块*之外无法引用它们。该约束保证这些内存类型的生命周期管理正确。

### 小结

本节介绍了 CroqTile 在异构计算环境中的并行方式，重点包括 `parallel-by` 块语法以及多级与子级并行的管理，还说明了 CroqTile 如何抽象内核启动并施加内存管理约束，从而有效利用并行硬件。
