## 概述
数据移动在 CroqTile 中被抽象为最复杂的语句形式。本节将介绍基本 *DMA 语句*结构及其产生的 future 变量。

## 数据移动语句

CroqTile 旨在应对多层次的数据移动，当前主要关注异构硬件之间以及跨内存层次的数据移动。在硬件术语中，这类移动称为**直接内存访问（Direct Memory Access，DMA）**。CroqTile 采用该术语，并将数据移动抽象为 **DMA 操作**。

### 基本语法

CroqTile 中的 DMA 语句可以是异步实体，即**异步 DMA**之后的代码可与 DMA 语句并行执行。对于*异步 DMA*，在使用 DMA 结果前必须进行显式同步。

为支持这些特性，语法组织如下：

```choreo
future = dma-op src-expr => dst-expr;
```

该语句定义 **future** 类型变量，出现在运算符 `=` 左侧。在 CroqTile 中，**future 既表示异步执行实例的句柄，也表示 DMA 结果**。因此，对于*同步 DMA*语句，可忽略 future 的定义。

**DMA 操作**右侧包括**操作类型**（`dma-op`）、操作源的**数据表达式**（`src-expr`）以及目标的*数据表达式*（`dst-expr`）。源与目标由符号 `=>` 分隔，表示数据流动方向。

### 同步与异步

**异步 DMA（Asynchronous DMA）**，或称 **Async-DMA**，亦称**非阻塞 DMA（Non-Blocking DMA）**，因为不会阻塞后续代码执行。反之，**同步 DMA（Synchronous DMA）**，或称 **Sync-DMA**，亦称**阻塞 DMA（Blocking DMA）**，因为 DMA 完成前代码不会继续。下文将交替使用这些术语。

在 CroqTile 中，**DMA 操作**上的 `.async` 后缀表示该操作为异步。示例如下：

```choreo
f0 = dma.copy.async data0 => shared; // non-blocking
f1 = dma.copy data1 => shared;       // blocking
...
wait f0;                             // synchronization
```

本例中，产生 future `f0` 的 DMA 操作为异步，程序可在后台传输数据的同时继续执行。产生 `f1` 的 DMA 操作为同步，即程序会暂停，直到数据从 `data1` 完全传送到 `shared` 内存。与此同时，`data0` 的拷贝可能仍在进行。

执行语句 `wait f0;` 会使程序检查 `data0` 是否已完全传送到 `shared` 内存，从而将主执行流与 `f0` 的异步 DMA 同步。注意，若未对 *Async-DMA* 进行等待，通常会导致意外的硬件问题，因此程序员必须确保代码中不存在未处理的「悬空 future」。


### 操作类型

CroqTile 的 DMA 语句是为支持现代硬件而设计的抽象。除简单线性内存拷贝外，**张量内存加速器（Tensor Memory Accelerator，TMA）**等高级硬件可在传输带形状数据的同时在传输过程中应用形状变换。CroqTile 在软件层将这些能力映射为不同的 **DMA 操作类型**。支持的操作包括：

- `dma.copy`：直接拷贝扁平内存。
- `dma.pad`：在传输数据的同时对带形状数据进行填充。
- `dma.transp`：在传输数据的同时对带形状数据进行转置。

填充、转置等细节作为 DMA 操作的参数编程。例如：

```choreo
global f32 [32, 16, 9] input;
dma.transp<0, 2, 1> input => shared;  // Result shape [32, 9, 16]
dma.pad<{1, 0, 3}, {0, 1, 2}, {0, 0, 0}, 0.1f> input => shared; // Result shape [33, 17, 14]
```

此处 **DMA 配置**由 `<>` 括起，不同操作的配置不同。特定平台（如 *CUDA/Cute*）的详细配置语法与限制以平台文档为准。

程序员应注意，除 `dma.copy` 外的 DMA 类型支持因平台而异。例如在 *CUDA/Cute* 上，可能是 TMA 的映射，或由多线程 load 指令编排而成。也可仅用软件或软硬件协作实现高级 DMA，但这些目前尚未支持。

### 数据表达式

此前我们仅使用简单的数据表达式，要么是已定义的 **spanned data**，要么是**存储位置（storage location）**。**存储位置**仅用于**目标缓冲区**声明，因为需要 CroqTile 编译器为其分配内存。示例如下：

```choreo
global f32 [32, 16] input;
shared f32 [32, 16] output;
dma.copy input => output;  // Copy to a user-declared buffer
dma.copy input => shared;  // Same, but to a compiler-allocated buffer
```

本例中第二次 DMA 的目标指定为存储位置 `shared`，需要编译器据此分配存储，因而与第一次 DMA 等价。实践中建议将存储位置作为目标，因为 CroqTile 可从 **DMA 语句**推断目标形状。

然而，实践中很少仅使用简单的**数据表达式**。下一节将介绍 **ChunkAt** 表达式，说明如何在数据表达式中实现分块（tiling）。

### Future 与 Wait

*Sync-DMA* 与 *Async-DMA* 均可用于定义 *future*。如前所述，*future* 作为异步操作的句柄，通常作为 `wait` 语句的参数以同步 *Async-DMA*。

CroqTile 允许 `wait` 语句接受多个 *future*。但若将来自**同步 DMA**的 *future* 用于 `wait` 会报错。示例如下：

```choreo
f0 = dma.copy.async input0 => shared;
f1 = dma.copy.async input1 => shared;
f2 = dma.copy input2 => shared;
...
wait f0, f1;  // Multiple wait
wait f2;      // Error: cannot wait on a sync-dma
```

在 CroqTile 中，*future* 的另一用途是获取关联 DMA 语句的目标缓冲区。这也是允许**同步 DMA**定义 future 的原因。CroqTile 为 *future* 变量提供两个内置成员函数：

- `.span`：获取 DMA 目标缓冲区的 **mdspan**。
- `.data`：获取 DMA 目标缓冲区的引用（spanned data）。

例如：

```choreo
f0 = dma.copy.async input0 => shared;
f1 = dma.copy input2 => shared;
...
local f32 [f0.span] buffer;
call device_kernel(f1.data, |f1.span|);
```

本例中使用 `.span` 获取 `f0` 目标缓冲区的形状，使用 `.data` 获取 `f1` 目标的 *spanned* 数据。此外，对 `f1` 目标缓冲区的 *mdspan* 使用 **ElementCount** 运算。**ElementCount** 返回 **mdspan** 中的元素个数，在许多场景下很有用。
