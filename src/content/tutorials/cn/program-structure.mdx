## 概述
本节将介绍 *Croktile-C++* 程序的基本结构及其相关术语。此外，我们还将给出一个并行的逐元素加法 *Croktile-C++* 程序，展示 Croktile 如何简化异构硬件上的数据编排。

## 构建 Croktile-C++ 程序
典型的 *Croktile-C++* 程序由多个部分组成，具体取决于目标平台。对于 Croktile 支持的平台——通常是利用异构并行硬件的编程环境——Croktile-C++ 程序一般包含三个部分：

* 设备程序（Device Program）
* 主机程序（Host Program）
* Tileflow 程序（Tileflow Program）

### 主机、设备与 Tileflow
下面的代码展示了一个面向 *CUDA/Cute* 的 *Croktile-C++* 程序：

```choreo
// Device program: typically runs on GPU/NPU
__device__ void device_function(...) {
  // High-performance device kernel implementation
}

// Tileflow program: orchestrating data movement
__co__ void choreo_function(...) {
  // ... choreo code ...
  device_function(...);
  // ...
}

// Host program: typically runs on CPU
void main() {
  // ... prepare data ...
  choreo_function(...);
  // ...
}
```

下面简要说明各部分：

**主机程序（Host Program）**

*主机程序*是 *Croktile-C++* 模块/程序的入口，也是 *Tileflow* 程序（Croktile 函数）的调用方。它使用标准 C++ 编写，在 CPU 上运行，并管理异构应用的整体工作流。

在简单的高性能内核实现中，程序员通常在主机程序中准备必要的数据，以调用 *Croktile 函数*，并处理其返回值以继续后续步骤。

**设备程序（Device Program）**

*设备程序*定义在目标设备上执行的重计算操作。在上面的示例中，设备函数带有 `__device__` 前缀，这是 *CUDA/Cute* 中的关键字，表示该函数仅在异构设备上运行。与 *主机程序*类似，任何设备程序在 Croktile 编译过程中都不会被修改。

**Tileflow 程序（Tileflow Program）**

熟悉 *CUDA/Cute* 的读者可能已经对主机程序和设备程序有所了解。然而，由 *Croktile 函数*（以 `__co__` 为前缀）构成的 *Tileflow 程序*才是 *Croktile-C++* 程序的核心。它在不同主机/设备之间，以及在同一设备内的不同存储层级之间编排数据移动。在典型工作流中，Tileflow 程序将数据移动到合适的存储位置（作为缓冲区），并调用 *设备程序*执行计算；工作完成后，再将结果移回主机。

### 转译与编译
Croktile 的编译过程通常包含三个主要步骤：**预处理**、**转译**与**目标编译**。为便于理解 Croktile-C++ 程序各部分如何协同工作，完整的编译工作流如下图所示：

![Croktile-Workflow](/tutorials/figures/compile-workflow.drawio.png)

如图所示，在 *预处理*之后，Croktile 会 *转译*（*源码到源码*的编译）*tileflow 程序*为目标代码，而用户提供的 *主机程序*与*设备程序*保持不变。*tileflow 程序*被 *转译*为主机与设备的源码，我们称之为 *Croktile 生成代码*。编译器再将用户代码与 *Croktile 生成代码*合并，执行 *目标编译*。该过程可产生多种输出，例如转译后的源码、工作脚本、目标模块、目标汇编以及可执行文件。

因此，Croktile 编译器充当端到端编译器，其中关键步骤是将 *tileflow 程序*转译为 *Croktile 生成代码*。

Croktile 编译的一个显著特性是同时支持：

- **单源编译模型（Single Source Compilation Model）**：类似 *CUDA/Cute*，目标编译器允许设备与主机程序位于同一源文件中进行目标编译。
- **分离源编译模型（Separate Source Compilation Model）**：类似 *OpenCL*，主机与设备代码必须分开编译。

上文所示代码自然支持 *单源编译模型*。若要支持 *分离源编译模型*，Croktile 需要用 `__cok__` 块包裹设备程序，如下所示：

```choreo
__cok__ {
  void device_function(...) { ... }
} // end of __cok__

__co__ void choreo_function(...) { ... }

void foo() { ... }
```

这是 *Croktile-OpenCL C++* 程序的代码结构。OpenCL 编译器要求设备程序（代码中的 `device_function`）与主机程序存放在不同文件中。`__cok__ {}` 包装使 Croktile 编译器能够正确处理用户提供的设备代码，并帮助 Croktile 从单一 Croktile 源文件中分离设备与主机代码，以适配不同的编译流程。因此，若在部分 Croktile 代码中遇到 `__cok__`，不必惊讶；这是为集成对 *分离源编译模型*的支持所必需的。

## 完整的 Croktile-Cute C++ 代码示例
下面是一个完整的 *Croktile-Cute C++* 代码示例，对两个长度与元素类型相同的数组执行逐元素加法：

```choreo
// Device Program
__device__ void kernel(int * a, int * b, int * c, int n) {
  for (int i = 0; i < n; ++i) c[i] = a[i] + b[i];
}

// Tileflow Program
__co__ s32 [6, 17, 128] ele_add(s32 [6, 17, 128] lhs, s32 [6, 17, 128] rhs) {
  s32 [lhs.span] output; // Use same shape as lhs

  // first `parallel` indicates the kernel launch
  parallel p by 6 {
    with index in [17, 4] { // Tiling factors
      foreach index {
        lhs_load = dma.copy lhs.chunkat(p, index) => local;
        rhs_load = dma.copy rhs.chunkat(p, index) => local;

        local s32 [lhs_load.span] l1_out;

        // Call kernel with loaded data
        call kernel(lhs_load.data, rhs_load.data, l1_out, |lhs_load.span|);

        // Store result back to output
        dma.copy l1_out => output.chunkat(p, index);
      }
    }
  }
  return output;
}

// Host Program
int main() {
  // Define data arrays
  choreo::s32 a[6][17][128] = {0};
  choreo::s32 b[6][17][128] = {0};

  // Fill arrays with data
  std::fill_n(&a[0][0][0], sizeof(a) / sizeof(a[0][0][0]), 1);
  std::fill_n(&b[0][0][0], sizeof(b) / sizeof(b[0][0][0]), 2);

  // Call Choreo function (data movement and device kernel execution)
  auto res = ele_add(choreo::make_spanview<3>(&a[0][0][0], {6, 17, 128}),
                     choreo::make_spanview<3>(&b[0][0][0], {6, 17, 128}));

  // Verification: check correctness of results
  for (size_t i = 0; i < res.shape()[0]; ++i)
    for (size_t j = 0; j < res.shape()[1]; ++j)
      for (size_t k = 0; k < res.shape()[2]; ++k)
        if (a[i][j][k] + b[i][j][k] != res[i][j][k]) {
          std::cerr << "result does not match.\n";
          abort();
        }

  std::cout << "Test Passed\n" << std::endl;
}
```

后续章节将解释该代码的各个部分。

### 主机程序——控制中心
如前所述，**主机程序**是 *Croktile-C++* 程序的入口，并充当控制中心。为便于阅读，再次列出相关代码：

```choreo
int main() {
  // Define data arrays
  choreo::s32 a[6][17][128] = {0};
  choreo::s32 b[6][17][128] = {0};

  // Fill arrays with data
  std::fill_n(&a[0][0][0], sizeof(a) / sizeof(a[0][0][0]), 1);
  std::fill_n(&b[0][0][0], sizeof(b) / sizeof(b[0][0][0]), 2);

  // Call Choreo function (data movement and device kernel execution)
  auto res = ele_add(choreo::make_spanview<3>(&a[0][0][0], {6, 17, 128}),
                     choreo::make_spanview<3>(&b[0][0][0], {6, 17, 128}));

  // Verification: check correctness of results
  for (size_t i = 0; i < res.shape()[0]; ++i)
    for (size_t j = 0; j < res.shape()[1]; ++j)
      for (size_t k = 0; k < res.shape()[2]; ++k)
        if (a[i][j][k] + b[i][j][k] != res[i][j][k]) {
          std::cerr << "result does not match.\n";
          abort();
        }

  std::cout << "Test Passed\n" << std::endl;
}
```

`main` 函数使用标准 C++ 编写，仅在使用 Croktile API 处有所不同。本程序中，我们先定义数组 `a` 与 `b` 并填入不同数值，再通过 API `croktile::make_spanview` 为数据附加形状信息。

`croktile::make_spanview` 的声明如下：

```cpp
template <size_t Rank, typename T>
spanned_view<T, Rank> make_spanview(T* ptr, std::initializer_list<size_t> init);
```

用法示例如下：

```cpp
choreo::make_spanview<3>(&a[0][0][0], {6, 17, 128});
```

该 API 对于连接 *主机代码*与 *Croktile 函数*至关重要。本质上，任何 Croktile 函数的输入缓冲区（称为 `spanned` 数据）都必须与其形状关联，从而使 Croktile 能在编译期与运行期保证形状安全。

**说明：**在 `initializer_list` 表示的形状中，**最高维**的数值排在最前。因此，形状 `{6, 17, 128}` 对应于类似 `a[6][17][128]` 的 C 多维数组。

在示例代码中，调用 Croktile 函数 `ele_add` 以并行计算逐元素和；随后，主机代码对结果缓冲区 `res` 进行校验。

一个重要细节是：Croktile 函数的输出类型为 `croktile::spanned_data`。与并不拥有其所指向数据缓冲区内存的 `croktile::spanned_view` 不同，`croktile::spanned_data` 拥有缓冲区，从而保证后续数据校验作用于有效内存。`croktile::spanned_view` 提供丰富 API，支持 C 风格数组下标访问，以及通过成员函数 `.shape()` 查询形状。

同样，在该形状数组中，最高维排在最前（本例中为 `res.shape()[0]`）。Croktile 采用「**最高维优先**」顺序，也称为「**行主序**」：第一维变化最慢。

### 设备程序：并行计算

**设备程序**定义将在 GPU 等目标设备上执行的计算内核。该内核对输入数据进行并行处理并产生输出。

代码再次列出如下：

```choreo
__device__ void kernel(int * a, int * b, int * c, int n) {
  for (int i = 0; i < n; ++i) c[i] = a[i] + b[i];
}
```

对于仅支持*分离源编程模型*的目标，代码可能需要包在 `__cok__ {}` 块内。等价的 *Croktile-OpenCL C++* 代码如下：

```choreo
__cok__ {
  extern "C" void kernel(int * a, int * b, int * c, int n) {
    for (int i = 0; i < n; ++i) c[i] = a[i] + b[i];
  }
} // end of __cok__
```

此处用 `extern "C"` 标注替代 *CUDA/Cute* 目标中使用的 `__device__` 关键字，因为 *OpenCL* 要求设备函数具有 C 链接。

Croktile 的设备编程模型因目标硬件及其支持特性而异。例如，部分私有目标允许使用向量化编程接口或*内建函数（intrinsic）*，以充分发挥并行目标硬件的计算能力。

程序员应注意，设备程序遵循*单程序多数据（SPMD）*范式：同一设备程序的多个实例并行执行，从而高效挖掘目标硬件上的数据级并行。然而，与传统 *CUDA/Cute* 程序不同，设备程序不负责管理数据移动——无论是主机与设备之间，还是设备内多个存储层级之间；这些任务由 *tileflow 程序*以更简单、更安全的方式编排。

### Tileflow 程序：编排数据移动

*Tileflow 程序*由 *Croktile 函数*组成。如前所述，它管理主机与目标设备之间的数据移动，并确保数据在不同存储位置之间被正确拷贝。

为便于阅读，再次回顾代码：

```choreo
__co__ s32 [6, 17, 128] ele_add(s32 [6, 17, 128] lhs, s32 [6, 17, 128] rhs) {
  s32 [lhs.span] output; // Use same shape as lhs

  // first `parallel` indicates the kernel launch
  parallel p by 6 {
    with index in [17, 4] { // Tiling factors
      foreach index {
        lhs_load = dma.copy lhs.chunkat(p, index) => local;
        rhs_load = dma.copy rhs.chunkat(p, index) => local;

        local s32 [lhs_load.span] l1_out;

        // Call kernel with loaded data
        call kernel(lhs_load.data, rhs_load.data, l1_out, |lhs_load.span|);

        // Store result back to output
        dma.copy l1_out => output.chunkat(p, index);
      }
    }
  }
  return output;
}
```

在这段代码中，以 `__co__` 为前缀的 Croktile 函数接受两个输入 `lhs` 与 `rhs`，形状均为 `[6, 17, 128]`，元素类型为 `s32`（有符号 32 位整数）；输出定义为与输入相同的形状与类型。

`parallel p by 6 {...}` 块表示其中封装的代码并行执行。具体而言，会同时运行六个该代码的实例，这意味着执行环境从主机过渡到设备。熟悉 *CUDA* 的读者可将此概念类比为内核启动：在设备上同时发起多个线程或进程以执行计算。

在 `parallel-by` 块内，`with-in` 块将符号 `index` 绑定到两个值 `17` 与 `4`。在 Croktile 中，`index` 被称为具有两个 `bounded` 值的 `bounded-ituple`，可用于 `foreach` 语句。（我们将在后续章节说明 `bounded` 类型。）

`foreach index {...}` 语句等价于以下 C 代码：

```cpp
for (int x = 0; x < 17; x++)  // assume 'x' represents the 1st element of 'index'
  for (int y = 0; y < 4; y++) { ... }  // and 'y' represents the 2nd element of 'index'
```

在 `foreach` 块内，`dma.copy` 语句描述数据如何移动。例如，考虑语句 `lhs_load = dma.copy lhs.chunkat(p, index) => local;`：

- 符号 `lhs_load` 称为该 DMA 操作的 **future**，其中包含 DMA 目标的相关信息。
- `dma.copy` 发起直接 DMA 数据传输，不改变数据的形状。`=>` 左侧表达式表示 DMA 源，右侧表示目标。
- 本例中目标指定为 `local` 缓冲区，由 Croktile 编译器自动分配。
- 源表达式 `lhs.chunkat(p, index)` 在 Croktile 中称为 **chunkat 表达式**。此处 `p` 与 `index` 是 `lhs` 的分块因子。鉴于 `lhs` 形状为 `[6, 17, 128]`，且 `p` 与 `index` 的上界分别为 `6`、`17` 与 `4`，数据块大小为 `1×1×32`（即 `6/6`、`17/17`、`128/4`）。每次迭代使用一个数据块作为源，具体块由当前 `p` 与 `index` 的值决定。例如，在并行线程 `1` 与迭代 `{16, 2}` 时，块的偏移为 `{1, 16, 2}`。

下图说明了这一点：

![Croktile-Structure-Chunkat](/tutorials/figures/chunkat-6-17-4.drawio.png)

借助 DMA 语句，从 `lhs` 分块得到的各个数据块会迭代且并行地从主机移动到设备的 `local` 存储。类似地，DMA 语句通过将 `rhs` 分小块移动到 `local` 存储来进行处理。

接下来，语句 `local s32 [lhs_load.span] l1_out;` 定义每个并行线程的缓冲区。注意其形状来自表达式 `lhs_load.span`，表示分块后的块；该缓冲区用于在随后的 `call` 语句中存放输出数据。`call` 语句调用名为 `kernel` 的*设备函数*执行计算；计算完成后，另一条 DMA 语句将输出数据从 `local` 缓冲区写回主机。这完成一次迭代。

在本代码中，每个并行线程运行 `17×4` 次迭代，每次迭代处理大小为 `1×1×32` 的数据块。*Croktile 程序*在所有 `6` 个并行线程完成迭代后结束，然后将输出缓冲区返回给其调用方——主机程序。

## 快速小结
现在您已了解 Croktile-C++ 程序由三部分组成，其中 *tileflow 程序*是核心；该部分在编译过程中被转译为目标源码。调用链通常从主机到 tileflow 程序，再从 tileflow 程序到设备代码。

您可能已经注意到，Croktile 不仅将 DMA 操作抽象为更高层语义，还将迭代与分块结合以便使用，从而使 Croktile 代码简洁且表达力强。在后续章节中，我们将更深入地探讨 Croktile 的语法与语义，以发掘其全部潜力。
