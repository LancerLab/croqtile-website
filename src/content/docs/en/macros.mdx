## Overview
Macros have long been a fundamental feature in C/C++ programming. To enable seamless manipulation of both C++ and *tileflow* programs, macro expansion is essential. This section demonstrates CroqTile's macro processing capabilities.

## Object-Like Macros
CroqTile preprocessor supports C/C++ **object-like macros** to connect both host code and tileflow code. The *object-like macro* performs pure text replacement and accepts no parameters. The below code gives an example:

```cpp
#define M 256
#define N 32
#define K 64

__co__ auto matmul(f32 [M, N] lhs, f32 [N, K] rhs) { /*...*/ }

void foo() {
  croqtile::f32 a[M][K];
  croqtile::f32 b[N][K];
  // ...
  auto res = matmul(croqtile::make_spanview<2>(a, {M, K}),
                    croqtile::make_spanview<2>(b, {N, K}));
}

```
In the code snippet, the inputs of croqtile function `matmul` are not dynamically shaped. Instead, CroqTile pre-processor substitutes `M`, `N`, `K` with values `256`, `32`, `64` ahead of CroqTile compilation, which produces tileflow function with statically shaped inputs. In this way, programmers makes different code consistent and easy to manipulate.

Note that, till now CroqTile does only support macros with parameters. It means code like:
```cpp
#define MAX(a, b) (((a) > (b)) ? (a) : (b))
```
would not work.

## Comments
CroqTile support C-style comments, either `/*...*/` or `//...`, leveraging the capability croqtile preprocessor has provided.

## Conditional Compilation
CroqTile also support conditional compilation which C/C++ programs used a lot. This includes `#if`/`#ifdef`/`#ifndef`/`#else`/`#endif`. The below code snippet showcases the usage:

```cpp
#define PATH0
// some host code
__co__ foo() {
#ifdef PATH0
// some code related to PATH0
#else
// other code
#endif
}

// host control
#ifdef PATH0
// ...
#else
// ...
#endif
```
In this way, it also make host and croqtile code be controlled within the same preprocess method.

Note that, the capability of croqtile preprocessor is still enhancing. But it is likely that we would not implement full C preprocessing support. CroqTile would not pick up existing C features unless people find it is necessary.

## Difference with C++ pre-processing
One important notices about croqtile preprocessing is that it is triggered much earlier than c++ preprocessing. The workflow of croqtile compilation is as shown below:

```
croqtile-preprocessing -> croqtile compilation -> c/c++ preprocessing -> c/c++ compilation
```

The primary target of croqtile preprocessing is to make host/device macros work as a whole, but such a workflow makes it possible sometimes different. From an implementation perspective, CroqTile pre-processor only substitute/conditionally-compile code inside the tileflow function, while leaving other pre-processing to the C++ preprocessor. That could restrict CroqTile pre-processing in a limited scope.

## Pre-defined Macros
To mimic a target native compilation, croqtile preprocess also takes the builtin macros from the target. For example, `__CUDA__` is globally defined to generate CUDA/Cute code, while `__CUDA_ARCH__` is only set for CUDA/Cute device code compilation.
Consequently, these macros can be utilized inside tileflow functions as well as the host code.
