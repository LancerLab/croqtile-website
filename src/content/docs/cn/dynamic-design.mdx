## 概述

本节介绍 CroqTile 中的*匿名维度（anonymous dimension）*与*符号维度（symbolic dimension）*，以及它们如何支撑某些系统所需的动态形状。

## 固定形状与动态形状

固定形状的数据常见于高性能计算内核，已知维度有利于积极优化。但某些场景需要处理运行期才确定的维度，即动态形状；这要求内核具备通用性以处理不同形状的输入，许多机器学习框架也强调这一点。

## 带**匿名维度**的 *mdspan*

CroqTile 支持取值未知的形状维度，与许多高层机器学习语言类似。最简单的方法是在 `mdspan` 中使用**匿名维度**：

```cpp
__co__ auto foo(s32 [?, 1, 2] input) { ... }
```

问号 `?` 表示编译期未知的正值；此处可假定任意正整数，但运行时必须提供实际值，如下例：

```cpp
void bar(int * data, int dim0) {
  foo(croqtile::make_spanview<3>(data, {dim0, 1, 2}));
}
```

本例中，`input` 的第一维由 `foo` 的*跨度*输入提供，其值来自 `bar` 的调用方在执行时传入。你可以假定该值来自配置文件、机器学习框架中的形状推导等。

*匿名维度*在此类场景下可以正常工作；但在某些情况下并不够用。

## 更进一步：**符号维度**

CroqTile 支持动态形状的另一种做法是使用**符号维度**。*符号维度*有名字，但同样在编译期未知。例如：

```cpp
__co__ auto Matmul(s32 [M, K] lhs, s32 [K, N] rhs) {
  int tile_m = 32, tile_n = 8, tile_k = 16;

  s32 [M / tile_m, K / tile_k] tiled_lhs;
  s32 [K / tile_k, N / tile_n] tiled_rhs;

  // ...
}
```

代码中为*跨度*输入的每一维赋予符号名（本例为 `M, K, N`）。与*匿名维度*相比，*符号维度*更清晰地描述形状之间的关系，代码直观、易维护；此外，还可利用额外信息提升安全性。下面深入说明。

## 更强的代码安全性

使用*符号维度*能提高安全性的原因在于，CroqTile 编译器可施加更全面的检查。例如，考虑 `Matmul` 函数的*匿名维度*版本：

```cpp
__co__ auto Matmul(s32 [?, ?] lhs, s32 [?, ?] rhs) {
  int tile_m = 32, tile_n = 8, tile_k = 16;

  s32 [lhs.span / {tile_m, tile_k}] tiled_lhs;  // 'mdspan' and 'ituple': rank must be same
  s32 [rhs.span / {tile_k, tile_n}] tiled_rhs;  // 'tiled_rhs' can not have a zero dimension

  // ...
}
```

CroqTile 在编译期与运行期都会做检查以保证安全：

- **编译期检查**：CroqTile 校验秩（rank）一致性。本例中 `lhs.span` 的秩为 `2`，与 `{tile_m, tile_k}` 的秩一致，因此无问题。
- **运行期检查**：CroqTile 生成代码在运行期校验维度。例如在 `tiled_lhs` 的声明中，其形状不能出现为 `0` 的维度，否则会产生无效的零大小缓冲区。CroqTile 的*转译（transpilation）*过程会生成如下*目标宿主代码*：

```cpp
void __croqtile_transpiled_Matmul(croqtile::span_view<2, croqtile::s32> lhs,
                                croqtile::span_view<2, croqtile::s32> rhs) {
  croqtile_assert(lhs.shape()[0] / 32 > 0,
                "the 0 dimension of 'lhs' may result in spanned data "
                "with a dimension value of 0.");
  // ...
}
```

此处若断言条件不满足，程序会中止；这种尽早检查有助于及时发现问题。

对于*符号维度*版本，CroqTile 编译器会在*目标宿主代码*中生成额外检查：

```cpp
void __croqtile_transpiled_Matmul(croqtile::span_view<2, croqtile::s32> lhs,
                                croqtile::span_view<2, croqtile::s32> rhs) {
  // additional check happens only when using symbolic dimensions
  croqtile_assert(lhs.shape()[1] == rhs.shape()[0],
                "dimension 'K' is not consistent.");

  croqtile_assert(lhs.shape()[0] / 32 > 0,
                "the 0 dimension of 'lhs' may result in spanned data "
                "with a dimension value of 0.");
  // ...
}
```

除*匿名维度*版本已有的检查外，*符号维度*版本还会校验 `lhs` 与 `rhs` 之间维度的一致性，从而更安全。

因此，建议程序员使用符号维度，不仅为了易用，也为了增强安全性。

## 代价与限制

为支持动态形状，CroqTile 编译器在编译期必须保守，这会限制部分编译期检查，并使某些检查改在运行期进行。尽管实现尽量把开销放在宿主侧，但与固定形状代码相比，这些检查**仍然更保守**。程序员应留意这一点，必要时可补充用户级断言以增强安全。

*匿名维度*与*符号维度*的一个重要限制是：它们**只能用于参数列表中的*跨度数据***。在 CroqTile 函数内部声明的缓冲区维度不能做成动态。例如下面代码会产生编译期错误：

```cpp
__co__ void foo() { f32 [M] d;} // compile-time error: can not applied to the spanned buffer
```

## 小结

本节讨论了 CroqTile 中的*匿名维度*与*符号维度*，它们支撑某些场景所需的动态形状。与*匿名维度*相比，*符号维度*通过额外的编译期检查增强安全性，在易用性与安全性上更值得推荐。
